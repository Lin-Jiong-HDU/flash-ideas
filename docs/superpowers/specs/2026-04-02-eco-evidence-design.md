# Eco Evidence（生态证据）设计文档

> 为 One4Earth Climate Change-Makers Challenge 2026 黑客松设计

---

## 产品定位

**名称：** Eco Evidence（生态证据）

**一句话：** 帮任何社区把日常的自然观察变成有法律效力的生态证据。

**核心洞察：** 现有的公民科学App（iNaturalist等）解决了"识别物种"的问题，但缺失把数据变成社区可以用来倡导改变的武器。我们不是科研工具，不是社交平台，不是环保组织对接系统——是一个**证据生成器**。

**目标用户：**
- 乡村/郊区学校的学生和教师（教育场景）
- 想要保护本地生态的居民（倡导场景）
- 长期关注本地生态的个人（证据储备场景）
- 任何想要记录社区生态变化的普通人

**核心价值：** 现有工具帮你"认识物种"，我们帮你"用数据说话"。它们是数据的**入口**，我们是数据的**出口**。

---

## 三层价值框架

| 层级 | 价值 | 说明 |
|------|------|------|
| **第一层（当下）** | 教育 + 连接 + 记忆 | 学生学会用数据描述自然；社区一起做有意义的事；长辈的记忆与数据并列 |
| **第二层（储备）** | 证据银行 | 持续积累，形成生态基线；即使现在不用，数据本身有长期价值 |
| **第三层（行动）** | 法律效力 | 当威胁出现时，EIA公众参与环节中有数据支撑的意见 ≠ 情感呼吁 |

**链条不依赖任何单一环节：**
- 如果没有开发威胁 → 第一层和第二层的价值独立存在
- 如果出现威胁 → 第三层价值激活，且有法律效力

## 法律机制：环境影响评估（EIA）

**这是 Eco Evidence 的核心法律基础。**

EIA 是 1992 年联合国里约地球峰会后推行的全球标准，100+ 国家有法定 EIA 制度。

**运作方式：**
1. 开发商提出建设项目
2. 法律要求做环境影响评估
3. 评估过程中有 **public consultation（公众参与）** 阶段
4. 任何公民/组织都可以提交意见和证据
5. 决策者**法律上有义务回应**有实质内容的异议（不能直接无视）

**关键区别：**
- 提交 "我不想要这个项目" → 情感诉求 → 可以被合法忽略
- 提交 "过去12个月我们在此区域记录到47个物种，包含X濒危物种" → **有数据的异议 → 法律要求回应**

**真实案例：** 英国大凤头蝾螈（Great Crested Newt）——社区居民通过生态调查记录到受保护物种，这些数据被提交到地方规划局，规划局法律上必须考虑这些数据，大量住房开发项目因此被推迟、修改或取消。

## 四个视角的体现

| 视角 | 如何体现 |
|------|----------|
| **Equity & Justice Lens** | 开源免费，面向资源较少的乡村社区；学生作为变革者；一个班级的学生可以生成与专业评估同等效力的报告 |
| **Practical Lens** | MVP 极简，4天可完成；只需智能手机和好奇心；无需昂贵设备 |
| **Two-Eyed Seeing Approach** | 每份报告都包含"Community Memory"板块——科学数据与代际知识并列，不是竞争，而是对话 |
| **Systems Approach** | 观察 → 可视化 → 报告 → 法律机制（EIA公众参与），连接到全球100+国家已有的保护生物多样性的法律制度 |

## 核心功能流程

```
┌──────────────────────────────────────────────────────────────┐
│                        Eco Evidence                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  STEP 1: 记录                                     │
│  ├── 拍照或从相册导入                                         │
│  ├── AI 识别物种（调用 iNaturalist API）                      │
│  └── 自动记录：时间 + 地点 + 物种                              │
│                                                              │
│  STEP 2: 聚合                                              │
│  ├── 选择/创建观察点                                          │
│  └── 同一地点的记录自动归类                                    │
│                                                              │
│  STEP 3: 可视化                                         │
│  ├── 时间线：物种出现/消失                                    │
│  ├── 物种数量变化曲线                                         │
│  └── 物种多样性指数趋势                                        │
│                                                              │
│  STEP 4: 报告生成 ← 核心差异化                     │
│  ├── 自动生成社区倡导报告                                      │
│  ├── 包含：数据 + 图表 + 社区记忆 + 建议措施                    │
│  └── 导出 PDF / 分享链接 / 用于EIA公众咨询                     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 报告智能分析

| 自动分析 | 说明 |
|----------|------|
| **下降警报** | 某物种连续 X 周未出现 → 标记为"可能消失" |
| **新增发现** | 首次记录的物种 → 标记为"新发现" |
| **季节性建议** | 根据季节推荐适合的本土植物种植 |
| **行动模板** | 根据物种类型（鸟类/昆虫/植物）匹配不同建议 |
| **社区记忆** | 提供开放式文本区，邀请社区长辈分享传统知识 |

### Two-Eyed Seeing 体现

报告中科学数据与传统知识并列：
- **科学视角**：物种清单、多样性指数、变化趋势
- **传统视角**：社区记忆板块，记录长辈眼中的"过去的样子"
- 两者不是竞争，而是对话——这正是 One4Earth 强调的 Two-Eyed Seeing

## 技术架构

### 技术选型

| 层级 | 技术 | 理由 |
|------|------|------|
| **前端框架** | Next.js 14 (App Router) | 全栈框架，前后端一体化，部署简单 |
| **UI** | TailwindCSS + shadcn/ui | 高质量组件，快速搭建专业 UI |
| **图表** | Recharts | React 生态，简单易用 |
| **后端** | Next.js API Routes | 与前端一体化 |
| **数据库** | Supabase (PostgreSQL) | 免费 tier 足够，Auth + DB + Storage 一站式 |
| **文件存储** | Supabase Storage | 与数据库集成 |
| **物种识别** | iNaturalist API | 免费开源，识别准确率高 |
| **报告生成** | @react-pdf/renderer | React 组件直接生成 PDF |

### 数据库设计

```sql
-- 观察点
CREATE TABLE locations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name TEXT NOT NULL,
  description TEXT,
  coordinates POINT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  owner_id UUID REFERENCES auth.users(id)
);

-- 观察记录
CREATE TABLE observations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  location_id UUID REFERENCES locations(id),
  species_name TEXT NOT NULL,
  species_common_name TEXT,
  confidence FLOAT,
  photo_url TEXT,
  observed_at TIMESTAMP DEFAULT NOW(),
  observer_name TEXT,
  notes TEXT
);

-- 报告
CREATE TABLE reports (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  location_id UUID REFERENCES locations(id),
  date_range_start DATE,
  date_range_end DATE,
  generated_at TIMESTAMP DEFAULT NOW(),
  pdf_url TEXT
);
```

### API 端点

```
POST   /api/observations        # 创建观察记录
GET    /api/observations        # 获取观察列表（支持按 location 筛选）
GET    /api/observations/:id    # 获取单条记录

POST   /api/locations           # 创建观察点
GET    /api/locations           # 获取观察点列表
GET    /api/locations/:id       # 获取单个观察点详情

POST   /api/identify            # 调用 iNaturalist API 识别物种
POST   /api/reports/generate    # 生成报告 PDF
GET    /api/reports/:id         # 获取报告
```

## 差异化总结

| 工具 | 他们做了什么 | 没做什么 | Eco Evidence |
|------|-------------|----------|--------------|
| **iNaturalist** | 物种识别 + 全球数据库 | ❌ 不生成报告 | 我们是数据的**出口**，可以整合它的API |
| **eBird** | 鸟类观察记录 + 统计 | ❌ 不生成报告 | 我们面向社区倡导，不是科研统计 |
| **GBIF** | 全球生物多样性数据聚合 | ❌ 面向科研，不面向社区 | 我们让普通人也能生成专业级报告 |
| **手动报告** | 无 | ❌ 耗时、需要专业知识 | 我们自动化数据收集和报告生成 |

**一句话区分：** 它们帮你认识物种，我们帮你把观察变成一份有说服力的文件。

## ⚠️ 我们不承诺什么

| 不要说 | 正确说法 |
|--------|----------|
| "我们的报告能阻止开发项目" | "我们让社区的声音有了数据支撑" |
| "环保组织会使用这些数据" | "数据在EIA公众咨询阶段有法律效力" |
| "社区数据等同于专业评估" | "社区数据可以作为触发更深入调查的证据" |

**诚实承认：** 社区生成的数据不带有与专业生态调查同等的法律效力。但它经常作为**触发器**——迫使当局更仔细地审视。
